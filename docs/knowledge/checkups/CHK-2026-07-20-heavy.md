### Cross-screen UX consistency

#### Material icon in an otherwise-Cupertino toolbar (`CHK-2026-07-20-heavy-5`)

- **Finding:** `syncStatusIconData` (`lib/slices/dashboard/ui/generic/sync_status_handler.dart:14-20`) unconditionally returns Material `Icons.*`. On iOS (`dashboard_page_ios.dart:202-205`) this sits in a toolbar of five otherwise-`CupertinoIcons.*` icons — the one Material glyph in an otherwise-Cupertino icon family, on the screen every user sees on launch. Nielsen heuristic #4 (Consistency and standards).
- **Debt quadrant:** prudent-inadvertent.
- **Disposition:** deferred — see ledger, deadline `2026-08-15`.
- **Graduation candidate:** no.

#### Inconsistent Android button hierarchy (`CHK-2026-07-20-heavy-6`)

- **Finding:** Android's primary-button hierarchy is inconsistent: most confirm actions use `FilledButton`, but the empty-dashboard "Create pact" CTA uses `ElevatedButton` (`dashboard_page_android.dart:122`), and `pact_detail_page_android.dart` mixes `ElevatedButton` (save note, 244), `OutlinedButton` (archive, 258), and `FilledButton` (stop, 274) for three same-file actions with no visible emphasis logic. iOS's equivalent CTAs consistently use `CupertinoButton.filled`. Nielsen heuristic #4.
- **Debt quadrant:** prudent-inadvertent.
- **Disposition:** deferred — see ledger, deadline `2026-09-01`.
- **Graduation candidate:** no.

#### No shared spacing or typography scale (`CHK-2026-07-20-heavy-7`)

- **Finding:** `lib/theme/` has no shared spacing or typography scale — only 3 shared widgets (`date_row_tile.dart`, `section_header.dart`, `status_badge.dart`). Every screen hardcodes `EdgeInsets`/`SizedBox` magic numbers (18 distinct `EdgeInsets` calls in `schedule_step_ios.dart` alone), and zero files under `lib/slices/` use `CupertinoTheme.of(context).textTheme` — all iOS text styling is ad hoc `TextStyle(fontSize: ...)` while Android consistently uses `Theme.of(context).textTheme.*`. This means there's no single place to fix contrast or dynamic-type issues on iOS (compounds the accessibility findings below).
- **Debt quadrant:** prudent-inadvertent.
- **Disposition:** deferred — see ledger, deadline `2026-10-14`.
- **Graduation candidate:** no.

### Accessibility

#### No semantic labels; sub-minimum tap targets on core flows (`CHK-2026-07-20-heavy-8`)

- **Finding:** The app has zero `Semantics`, `tooltip`, or `semanticLabel` usage anywhere under `lib/` (confirmed via project-wide grep) — every accessibility gap below is a symptom of this one systemic absence. Concretely: unlabeled `IconButton`s on the dashboard app bar (`dashboard_page_android.dart:169-192`, 4 buttons) and on pact creation/edit/detail close-and-edit controls; unlabeled destructive "remove slot" buttons (`slot_schedule_editor.dart:194,263`, `schedule_step_android.dart:185,276,341`); `OptionTile` (`option_tile.dart:27-46`) has no selection-state semantics. Compounding this, several core-flow tap targets sit well below the 44pt/48dp platform minimums: the wizard step indicator (`wizard_step_indicator.dart:27-46`, ~4px tall, present on every pact creation/edit screen), the "remove slot" icon buttons (~20×20), the weekday toggle chips (`slot_schedule_editor.dart:339-364`, ~24-26px tall, no selection semantics either), and the dashboard calendar-strip day circles (`dashboard_body.dart:116-129`, 40×40). Together these block screen-reader users on the app's most-used flows (dashboard, pact creation) and add friction for anyone with limited dexterity.
- **Debt quadrant:** reckless-inadvertent — not a deliberate choice to skip accessibility, but a mistake now clearly understood and severe enough (blocks screen-reader users on core flows) to warrant a shorter deadline than a typical hindsight design flaw.
- **Disposition:** deferred — see ledger, deadline `2026-08-15`.
- **Graduation candidate:** yes — `flutter_test`'s built-in accessibility guideline matchers (`meetsGuideline(textContrastGuideline)`, `meetsGuideline(androidTapTargetGuideline)`, `meetsGuideline(iOSTapTargetGuideline)`) need no new dependency and could be wired into the existing widget-test suite to catch tap-target and contrast regressions mechanically (HAB-143 territory).

#### Colour contrast and text-scaling gaps (`CHK-2026-07-20-heavy-9`)

- **Finding:** `CupertinoColors.systemGrey` (~2.85:1 on white, fails WCAG AA's 4.5:1 for normal text) is used repeatedly for body/value/header text at 12-16px across `pact_detail_page_ios.dart` (11 sites), `pact_timeline_page_ios.dart` (6 sites), the shared `wizard_style.dart` hint/label style (used by every wizard step on both platforms), and `showup_detail_page_ios.dart`. `DateRowTile`'s default `valueColor` (`lib/theme/widgets/date_row_tile.dart:15`) is `Colors.grey` (~2.3:1) — currently unused by any caller (all pass an explicit color) but a latent trap for a future one. Separately, the dashboard's 40×40 day-of-month circle (`dashboard_body.dart:122-129`) has no `FittedBox` around the digit, so it will visually overflow at large system text-scale settings. No `textScaleFactor`/`MediaQuery.textScaler` overrides fight the user's system preference anywhere (checked — this is a positive finding).
- **Debt quadrant:** prudent-inadvertent.
- **Disposition:** deferred — see ledger, deadline `2026-09-15`.
- **Graduation candidate:** yes — same `meetsGuideline(textContrastGuideline)` matcher as above covers this too; one wiring effort could close both accessibility findings' graduation candidates together.
