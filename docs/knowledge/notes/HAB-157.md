**Remember to register new integration scenario files in test_runner.dart**

Adding a new `integration_test/*_flow_test.dart` file makes it runnable standalone, but it will silently be skipped by the combined suite (`flutter test integration_test/test_runner.dart` / CI's `run-scenarios` job) unless it's also imported and called from `test_runner.dart`. This was missed when `reminder_locale_flow_test.dart` was first added during implementation — caught only after the user flagged it post-PR. Add the `import '...' as <alias>;` and the `<alias>.main();` call (both alphabetically placed) as part of the same commit that adds the scenario file, not as a follow-up.

**Prefer less nested code — consider adding this to the implement skill**

User feedback during this PR's review: reducing nesting (e.g. early-returns instead of if/else pyramids, guard clauses) makes code easier to read. Concretely demonstrated in this PR — `applyLanguageSelection`'s `if (...) { ... } else { ... }` had an `else` block that was dead weight (the `if` branch always returned), so flattening it to two sequential guard-clause-style blocks was a strict readability win with identical behavior. Reference video on the general principle: https://www.youtube.com/watch?v=CFRhGnuXG-4. Candidate action: add a "prefer flat control flow / early returns over nested if-else" guideline to `skills/build/implement/SKILL.md` or `docs/CODE_STYLE.md` so this is checked proactively during implementation rather than caught in review. Worth raising at the next `/debrief`.

Broader "never-nest" principle to fold into the same guideline (not just the dead-else case caught here):
- **Guard clauses over wrapping conditionals.** If a function body is one big `if (validCase) { ...everything... }`, invert it to `if (!validCase) return;` followed by unindented code. Reserve nesting for cases where both branches carry equal, ongoing weight.
- **Extract nested blocks into well-named functions** once nesting hits ~3 levels (e.g. `for` → `if` → `try`) rather than growing it further — a function call reads as one unit of thought; a deeply nested block doesn't.
- **Combine chained conditions instead of stacking `if`s** (`if (a) { if (b) { ... } }` → `if (a && b) { ... }`) when the two checks are logically one gate, not two independent decisions worth naming separately.
- **Loops:** prefer `continue`/`break` guard clauses at the top of a loop body over wrapping the whole body in `if (shouldProcess(item)) { ... }`.
- Net effect the video argues for: nesting depth is a readability cost independent of line count — two 4-line flat functions beat one 8-line function nested three deep, even though the nested version isn't otherwise "worse" by size or complexity metrics.
