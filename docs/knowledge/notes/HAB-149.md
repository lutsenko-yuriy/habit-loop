# HAB-149: About / Feedback screen

## Notes

### 2026-07-04

**Firebase RC housekeeping creates approval-wait latency**

Pushing new Remote Config flags (e.g. `about_screen_enabled`, `showup_redemption_enabled`) required several round-trips: fetch current config, draft the update, wait for user approval, push. Most of the elapsed time was waiting for the user to review and confirm the change before the push was made.

Possible improvements to discuss at debrief:
- **Pre-approve RC pushes in the workflow**: the plan skill already lists the flag name and default value — if the user approves the plan, that implicitly approves adding the flag to Firebase RC. The scenarios or WU0 step could push it automatically without a separate approval gate.
- **Dedicated RC sync step in `/ship`**: after merge, `/ship` could diff `remote_config_defaults.dart` against the live RC template and push any missing flags automatically.
- **RC flags as part of the CHANGELOG entry**: require that any new flag is pushed to Firebase RC before the PR is merged (checked via a CI step or checklist item), removing the need for a separate housekeeping session.

### 2026-07-04

**WU0 stub pattern: unused imports and dead-code from nullable harness variable**

The initial stub file failed CI `flutter analyze` with two errors: unused imports (`app_providers.dart`, `fake_remote_config_service.dart`) added speculatively for future test code, and a `dead_code` warning from `AppHarness? h` + `tearDown(() => h?.dispose())` — the analyzer correctly detected that `h` is always null in stub bodies.

Fix applied: remove speculative imports (add them per-WU during `implement`), replace the nullable harness variable + tearDown with a `// TODO:` comment for `implement` to restore.

Rule for future WU0 stubs: only import what is actually referenced in the stub file at commit time; leave harness lifecycle as a TODO comment rather than dead nullable code.

### 2026-07-05

**Licences screen: "like the rest of the app" analogy failed — needs a different approach**

Multiple iterations were spent trying to make the licences screen look native (themed `LicensePage`, custom `CupertinoLicensesPage` with `CustomScrollView` + `CupertinoSliverNavigationBar`, explicit text styles, background colours). Each iteration satisfied part of the request but not the whole. The analogy "like the rest of the app" was used as guidance but was too vague to converge on a result the user was happy with.

Outcome: the Licences entry point was removed from the About screen. The `CupertinoLicensesPage` file was also removed as orphaned code.

For the WU that reintroduces the licences screen: gather a concrete visual reference (screenshot or specific app) before implementing, rather than using "matches the rest of the app" as the spec. The ambiguity cost several iteration cycles.

### 2026-07-05

**Remove empty TODO scenario stubs before or at ship, not after**

SC-04 (`licences_row_opens_licence_page`) was left as an empty `// TODO:` stub in the scenario file because the Licences row was removed mid-WU1. It was only deleted at the start of WU2 when it was obvious it would never be filled in.

Rule: if a scenario stub has no remaining `TODO` steps to implement by the time its WU ships (because the feature was dropped or deferred), delete it before committing — don't carry dead stubs into main.

### 2026-07-05

**Fix review findings immediately, don't defer**

Both findings from the WU2 reviews (WidgetRef in application layer, silent launchUrl failure) were small and fixable in under 5 minutes. They should have been fixed before posting the review summary, not after the user had to ask. Rule: if a review or audit finding can be resolved in the same session without design discussion, fix it immediately and note the fix in the structured summary rather than leaving it as an open comment.
