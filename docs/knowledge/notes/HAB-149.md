# HAB-149: About / Feedback screen

## Notes

### 2026-07-04

**Firebase RC housekeeping creates approval-wait latency**

Pushing new Remote Config flags (e.g. `about_screen_enabled`, `showup_redemption_enabled`) required several round-trips: fetch current config, draft the update, wait for user approval, push. Most of the elapsed time was waiting for the user to review and confirm the change before the push was made.

Possible improvements to discuss at debrief:
- **Pre-approve RC pushes in the workflow**: the plan skill already lists the flag name and default value — if the user approves the plan, that implicitly approves adding the flag to Firebase RC. The scenarios or WU0 step could push it automatically without a separate approval gate.
- **Dedicated RC sync step in `/ship`**: after merge, `/ship` could diff `remote_config_defaults.dart` against the live RC template and push any missing flags automatically.
- **RC flags as part of the CHANGELOG entry**: require that any new flag is pushed to Firebase RC before the PR is merged (checked via a CI step or checklist item), removing the need for a separate housekeeping session.

### 2026-07-04

**WU0 stub pattern: unused imports and dead-code from nullable harness variable**

The initial stub file failed CI `flutter analyze` with two errors: unused imports (`app_providers.dart`, `fake_remote_config_service.dart`) added speculatively for future test code, and a `dead_code` warning from `AppHarness? h` + `tearDown(() => h?.dispose())` — the analyzer correctly detected that `h` is always null in stub bodies.

Fix applied: remove speculative imports (add them per-WU during `implement`), replace the nullable harness variable + tearDown with a `// TODO:` comment for `implement` to restore.

Rule for future WU0 stubs: only import what is actually referenced in the stub file at commit time; leave harness lifecycle as a TODO comment rather than dead nullable code.
