# HAB-167: Add TestFlight distribution alongside Firebase App Distribution

## Notes

### 2026-07-14

**WU scope summary (plan approved)**

- **WU1 — `feature/HAB-167-WU1-testflight-tooling`** (~130 LoC): the mechanically-verifiable half. Adds `scripts/appstore/testflight_upload.sh` (mirrors `scripts/firebase/distribute.sh`, uses `xcrun altool --upload-app` with an App Store Connect API key), a `distribute_testflight` flag in `scripts/ci/dispatch_plan.py` (mirrors `distribute_ios` gating), unit tests in `scripts/ci/tests/test_dispatch_plan.py`, `ITSAppUsesNonExemptEncryption = false` in `ios/Runner/Info.plist` (avoids the export-compliance prompt stalling the first upload), and doc stubs (`docs/VERSIONING.md`, `docs/ARCHITECTURE.md`, `docs/CHANGELOG.md` tagged `[wip]`). No CI wiring yet — pure unit-testable plumbing.
- **WU2 — `feature/HAB-167-WU2-ci-wiring`** (~150 LoC): wires it all into `.github/workflows/ci.yml` — `build-ios` gains a second `xcodebuild -exportArchive` (same `.xcarchive`, `method=app-store` this time, needs a new app-store provisioning profile) producing an `ios-appstore` artifact; a new isolated `distribute-testflight` job (`macos-15`) uploads it. Final WU, tagged `[ci]`. Validated live via `workflow_dispatch` on the feature branch (a `[ci]` entry sets `should_build=false` on normal merge, so this is the only way to exercise the real upload before shipping).

**Key architectural decision: signing mismatch**

FAD needs an **ad-hoc** IPA; TestFlight needs an **app-store**-signed IPA. Same archive, two exports (`xcodebuild -exportArchive` twice from one `.xcarchive`). Same Apple Distribution cert reused; only a new app-store provisioning profile is needed (`IOS_APPSTORE_PROVISIONING_PROFILE` secret).

**Key architectural decision: job isolation**

Went with **Option B** (separate `distribute-testflight` job, kept out of `version-tag`'s `needs`) over Option A (`continue-on-error` step inside `build-ios`). User's call: priority for this initial rollout is validating whether the whole approach works at all, so full isolation from FAD/tagging beats saving one extra `macos-15` runner. May revisit if runner-minute/cost limits become a problem later.

**Tooling correction to the ticket**

The ticket's original preference named `xcrun altool`/`notarytool`. `notarytool` is for macOS app notarization, not iOS TestFlight upload — not applicable here. The correct direct-API tool is `altool --upload-app`, which does satisfy the ticket's "prefer direct App Store Connect API over fastlane" preference.

**New secrets needed (one-time manual setup, not code)**

`APP_STORE_CONNECT_API_KEY_P8`, `APP_STORE_CONNECT_KEY_ID`, `APP_STORE_CONNECT_ISSUER_ID`, `IOS_APPSTORE_PROVISIONING_PROFILE`. Reuses existing `IOS_CERTIFICATE_P12`, `IOS_CERTIFICATE_PASSWORD`, `IOS_TEAM_ID`. Prerequisite: app record must exist in App Store Connect for `com.habitloop.habitLoop` before WU2's live validation.

### 2026-07-14

**Naming clarity — distribute flags/jobs ambiguous with a third destination**

`distribute_android` / `distribute_ios` (dispatch_plan.py flags, mirrored as CI job IDs) were clear when there were only two platforms. Adding `distribute_testflight` alongside them breaks the pattern — one axis is platform, the other is destination, so at a glance it's not obvious `distribute_ios` and `distribute_testflight` are peers. Proposed rename: `distribute_android_firebase`, `distribute_ios_firebase`, `distribute_ios_testflight`. Deferred to WU2, since that's when the CI job IDs (not just the Python flags) are touched and the `distribute-testflight` job is actually created.

**General takeaway for debrief:** applies to code in general, not just this naming case. Once the standard requirements are met — code works, is performant, is clean — also weight **glanceability**: a reader should be able to infer purpose from names/structure alone, without opening the implementation, even after more variants get added later. Not a precondition, but a real property to optimize for once the others are satisfied. Consider codifying this in `docs/CODE_STYLE.md`.
