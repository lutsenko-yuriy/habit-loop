# HAB-167: Add TestFlight distribution alongside Firebase App Distribution

## Notes

### 2026-07-14

**WU scope summary (plan approved)**

- **WU1 — `feature/HAB-167-WU1-testflight-tooling`** (~130 LoC): the mechanically-verifiable half. Adds `scripts/appstore/testflight_upload.sh` (mirrors `scripts/firebase/distribute.sh`, uses `xcrun altool --upload-app` with an App Store Connect API key), a `distribute_testflight` flag in `scripts/ci/dispatch_plan.py` (mirrors `distribute_ios` gating), unit tests in `scripts/ci/tests/test_dispatch_plan.py`, `ITSAppUsesNonExemptEncryption = false` in `ios/Runner/Info.plist` (avoids the export-compliance prompt stalling the first upload), and doc stubs (`docs/VERSIONING.md`, `docs/ARCHITECTURE.md`, `docs/CHANGELOG.md` tagged `[wip]`). No CI wiring yet — pure unit-testable plumbing.
- **WU2 — `feature/HAB-167-WU2-ci-wiring`** (~150 LoC): wires it all into `.github/workflows/ci.yml` — `build-ios` gains a second `xcodebuild -exportArchive` (same `.xcarchive`, `method=app-store` this time, needs a new app-store provisioning profile) producing an `ios-appstore` artifact; a new isolated `distribute-testflight` job (`macos-15`) uploads it. Final WU, tagged `[ci]`. Validated live via `workflow_dispatch` on the feature branch (a `[ci]` entry sets `should_build=false` on normal merge, so this is the only way to exercise the real upload before shipping).

**Key architectural decision: signing mismatch**

FAD needs an **ad-hoc** IPA; TestFlight needs an **app-store**-signed IPA. Same archive, two exports (`xcodebuild -exportArchive` twice from one `.xcarchive`). Same Apple Distribution cert reused; only a new app-store provisioning profile is needed (`IOS_APPSTORE_PROVISIONING_PROFILE` secret).

**Key architectural decision: job isolation**

Went with **Option B** (separate `distribute-testflight` job, kept out of `version-tag`'s `needs`) over Option A (`continue-on-error` step inside `build-ios`). User's call: priority for this initial rollout is validating whether the whole approach works at all, so full isolation from FAD/tagging beats saving one extra `macos-15` runner. May revisit if runner-minute/cost limits become a problem later.

**Tooling correction to the ticket**

The ticket's original preference named `xcrun altool`/`notarytool`. `notarytool` is for macOS app notarization, not iOS TestFlight upload — not applicable here. The correct direct-API tool is `altool --upload-app`, which does satisfy the ticket's "prefer direct App Store Connect API over fastlane" preference.

**New secrets needed (one-time manual setup, not code)**

`APP_STORE_CONNECT_API_KEY_P8`, `APP_STORE_CONNECT_KEY_ID`, `APP_STORE_CONNECT_ISSUER_ID`, `IOS_APPSTORE_PROVISIONING_PROFILE`. Reuses existing `IOS_CERTIFICATE_P12`, `IOS_CERTIFICATE_PASSWORD`, `IOS_TEAM_ID`. Prerequisite: app record must exist in App Store Connect for `com.habitloop.habitLoop` before WU2's live validation.

### 2026-07-14

**Naming clarity — distribute flags/jobs ambiguous with a third destination**

`distribute_android` / `distribute_ios` (dispatch_plan.py flags, mirrored as CI job IDs) were clear when there were only two platforms. Adding `distribute_testflight` alongside them breaks the pattern — one axis is platform, the other is destination, so at a glance it's not obvious `distribute_ios` and `distribute_testflight` are peers. Proposed rename: `distribute_android_firebase`, `distribute_ios_firebase`, `distribute_ios_testflight`. Deferred to WU2, since that's when the CI job IDs (not just the Python flags) are touched and the `distribute-testflight` job is actually created.

**General takeaway for debrief:** applies to code in general, not just this naming case. Once the standard requirements are met — code works, is performant, is clean — also weight **glanceability**: a reader should be able to infer purpose from names/structure alone, without opening the implementation, even after more variants get added later. Not a precondition, but a real property to optimize for once the others are satisfied. Consider codifying this in `docs/CODE_STYLE.md`.

### 2026-07-14 (continued, WU2 live validation)

**Bug found and fixed via live validation: API key not base64-decoded**

First real `workflow_dispatch` upload attempt failed: `altool` reported "does not contain a valid authentication key" for the staged `.p8` file. Root cause: `testflight_upload.sh` wrote `APP_STORE_CONNECT_API_KEY_P8` to disk verbatim (`printf '%s' ... > "$KEY_FILE"`), but the secret is base64-encoded — matching this repo's established convention for file-content secrets (`IOS_APPSTORE_PROVISIONING_PROFILE`, `FIREBASE_OPTIONS_DART`, etc. all use `echo "$VAR" | base64 --decode > file`), and matching what the plan and the walkthrough given to the user both specified. Fixed to `echo "$APP_STORE_CONNECT_API_KEY_P8" | base64 --decode > "$KEY_FILE"`.

**Takeaway:** this is exactly the kind of bug that only live validation (not unit tests, not code review) catches — the plan flagged this risk category upfront ("expect scope to balloon on first real upload"). Re-confirms the mitigation strategy (merge WU1 standalone, validate WU2 live via `workflow_dispatch` before merge) was the right call.

**Also added mid-review (user request, not in original plan):** two independent `workflow_dispatch` toggles, `distribute_firebase` and `distribute_testflight`, replacing the old single `deploy` boolean — lets either distribution channel be validated alone (e.g. TestFlight-only, without spamming Firebase testers with a validation build). Also clarified in the input descriptions and `docs/VERSIONING.md` that Firebase covers Android + iOS while TestFlight is iOS-only.

### 2026-07-14 (continued, WU2 live validation — new blocker found)

**Second live-validation finding: Apple now requires the iOS 26 SDK for App Store Connect uploads**

After the base64 fix above, a re-run got past authentication and actually reached Apple's validation step — then failed with:

> Validation failed (409) SDK version issue. This app was built with the iOS 18.5 SDK. All iOS and iPadOS apps must be built with the iOS 26 SDK or later, included in Xcode 26 or later, in order to be uploaded to App Store Connect or submitted for distribution.

Root cause confirmed via web search: `build-ios` runs on `runs-on: macos-15`, and GitHub's `macos-15` runner image does **not** carry Xcode 26 (held back due to on-image storage constraints, per `actions/runner-images` maintainers). Xcode 26 is only available on the `macos-26` runner image, GA since 2026-02-26.

**Why this wasn't caught earlier:** this is exactly the class of risk the plan flagged upfront ("expect scope to balloon on first real upload... only possible against App Store Connect, no way to catch pre-emptively"). Firebase App Distribution has no SDK-version gate, so the ad-hoc path never surfaced this.

**Open decision, deliberately not made solo overnight:** switching `build-ios`'s `runs-on` to `macos-26` would fix the TestFlight path, but that job also produces the existing ad-hoc/Firebase export — changing the whole job's toolchain (Xcode 16.4 → 26.x) risks behavior differences in the path that already works in production (CocoaPods, code signing, Flutter build all on a much newer Xcode). Options to weigh with the user:
1. Move only `build-ios` to `macos-26` (simplest, but couples FAD's toolchain to TestFlight's requirement).
2. Split into two jobs: keep ad-hoc export on `macos-15` (unchanged), add a second `macos-26` job that re-archives+exports app-store-only for TestFlight (more isolated, but archives twice — more build time/cost, and two build outputs from two different Xcode versions could diverge in edge cases).
3. Treat this as a follow-up ticket (per the plan's own mitigation clause) rather than blocking this PR further tonight.

**Status:** paused here pending user input — did not merge, did not switch runner images unilaterally.

### 2026-07-15 (decision: option 1 chosen)

User chose option 1 — moved `build-ios`'s `runs-on` from `macos-15` to `macos-26` entirely, rather than splitting into two jobs. `distribute-testflight` stays on `macos-15` (upload-only, no SDK dependency). Accepted risk: the existing ad-hoc/Firebase export path now also runs on the newer Xcode toolchain, untested in that combination until the next live validation run. Documented in `docs/VERSIONING.md`.

Explicitly a trial, not a final call: if `macos-26` breaks the ad-hoc/Firebase export path (signing, CocoaPods, or Flutter build differences), fall back to option 2 (split into two jobs) rather than persisting with option 1.
