## Debrief summary

### 2026-07-11

**What went well**
- Overall a smooth ticket; user was happy with the outcome and eager to see it validated in real day-to-day use.
- Turning a vague felt sense ("this doc is patchy") into precise, structured wording — the dialog-principles.md rewrite was called out specifically as landing well.

**What was hard or surprising**
- The hard part for the user wasn't scope or process, it was articulating precisely what they wanted the docs to end up looking like. Drafting concrete candidate wording for them to react to (rather than asking them to originate it) was what unblocked it — a real, repeatable pattern, not a one-off.

**What to change**
- `/note`-created files can get orphaned: this ticket's own knowledge note wasn't staged before `/ship` ran, survived a branch switch untracked, and only surfaced later by chance. Fixed: `ship/SKILL.md` step 6 now explicitly checks for `docs/knowledge/notes/HAB-XX*.md` before committing.
- Self-audit prompted by the user: the "draft concrete wording for the user to react to" pattern is in tension with `dialog-principles.md`'s "mirror, don't lead" (present alternatives, don't hand over one finished answer) whenever it happens unprompted rather than in direct response to an explicit "any ideas?" ask. Fixed: added a bullet to "Mirror, don't lead" — default to 2 short alternatives instead of one polished draft when originating language unprompted.
- Follow-on naming question raised (not resolved here): should "principles" be reframed as "guidelines," since every one of them has legitimate observed exceptions in practice? Tracked as HAB-160, deliberately scoped out of this debrief's file-change approval.
