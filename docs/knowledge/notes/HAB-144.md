**Step 2 survey findings — existing practices for periodic non-mechanical quality checkups**

Since this ticket is about engineering process rather than a product feature, the survey covered dev-practice precedent rather than consumer apps:

- Recurring cadence (monthly/quarterly) is the standard recommendation for technical debt review — without a cadence, debt tracking "becomes another abandoned initiative." [Codacy](https://blog.codacy.com/complete-guide-to-technical-debt-tracking-for-engineering-leaders)
- Fowler's **Technical Debt Quadrant** is the standard lens for classifying findings once found; most checkup findings will land in "prudent-inadvertent" (discovered only in hindsight). [Fowler](https://martinfowler.com/bliki/TechnicalDebtQuadrant.html)
- **Cross-screen UX consistency / accessibility**: NN/g's heuristic evaluation method (3–5 evaluators, Nielsen's 10 heuristics — "Consistency and standards" is #4) is the closest analog. [NN/g](https://www.nngroup.com/articles/how-to-conduct-a-heuristic-evaluation/)
- **Accessibility**: common guidance is automated scans continuously + a manual audit quarterly/annually for depth. [TheWCAG](https://www.thewcag.com/accessibility-audit-guide)
- **Glossary/naming drift**: DDD literature confirms drift is expected as implementation solidifies; the glossary itself needs periodic review, not just a one-time write. [Fowler: Ubiquitous Language](https://martinfowler.com/bliki/UbiquitousLanguage.html)
- **Feature-flag lifecycle**: closest real precedent to "monthly + on-demand" — teams block dedicated recurring time (weekly "Flag Friday" or quarterly audits) because stale flags compound. Assigning an explicit expiration/review-by date per flag is standard — directly analogous to the "next-due date" idea. [Statsig](https://www.statsig.com/perspectives/tips-for-unused-feature-flag-clean-up) · [FlagShark](https://flagshark.com/blog/feature-flag-lifecycle-creation-cleanup-5-stages/)
- **Doc-reality drift**: "docs-as-code" staleness detection assumes CI/mechanical enforcement, which doesn't fit a solo-dev monthly cadence and is out of scope here anyway. [Docsie](https://www.docsie.io/blog/glossary/documentation-drift/)
- **Unused functionality**: product-analytics practice defines a "kill condition" up front per feature (retire if usage doesn't cross X by date Y) rather than discovering staleness after the fact — doesn't retroactively help already-shipped features but worth considering for new ones. [Userpilot](https://userpilot.com/blog/product-feature-analysis/)
- **Tracking findings without a full ticket**: the closest precedent is a lightweight "technical debt register" (categorized by code area, with problem/fix/cost) — notably, that source recommends *against* hard deadlines, favoring "fix when back in that code area" instead. This is the opposite of the deadline approach chosen for this ticket — a deliberate deviation, since a solo dev has no natural "back in that code area" trigger. [Mark Heath](https://markheath.net/post/technical-debt-register)
- Feature-flag audits are the one place precedent *does* use a hard per-item review-by date without a ticket — reinforcing that pattern as the right fit here.
- **Cyclomatic complexity** (Discovery outcome example) already has an off-the-shelf Dart/Flutter tool — `dart_code_metrics`/DCM, configurable in `analysis_options.yaml` — confirming this dimension really could graduate to CI later. [DCM](https://dcm.dev/docs/metrics/function/cyclomatic-complexity/)

**Cite sources next to the decision they support, not just in a batched list**

User's earlier ask (above) to cite sources wasn't fully satisfied by dumping links in one survey block — the first draft of ADR-0003 put all context/citations in one paragraph, separated from the specific decision points they actually justify. Revised so each decision bullet in the ADR carries its own inline citation, and added a dedicated paragraph on using Fowler's Technical Debt Quadrant to classify findings (a survey finding that hadn't made it into the Decision section at all in the first draft). Applies to ADRs and any future write-up with multiple distinct claims: citation placement should be local to the claim, not centralized.
