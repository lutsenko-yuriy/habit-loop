**Run tests/scenarios before merging, not just before opening the PR — but scope this to checkup-findings-fix tickets specifically.** WU3's icon-family fix (`syncStatusIconData` → Material/Cupertino split) broke two integration tests (`sync_on_login_flow_test.dart`, `fresh_install_flow_test.dart`) that hardcoded Material icon assertions against the iOS dashboard — they only ever passed on iOS by coincidence, since the pre-fix bug made iOS render a Material icon too. `flutter test` (run during WU3's own validation) doesn't cover `integration_test/`, since those require a real device/simulator — so the regression wasn't caught until manually running the integration suite after WU3 had already merged, well into WU4. Running `/run-scenarios` before every merge on every ticket would be wasteful (computing cost) — the risk it guards against is specific to tickets like this one: a bundle of independent WUs touching unrelated, cross-cutting areas of the app (checkup-findings cleanup tickets, e.g. HAB-187/HAB-188's pattern), where one WU's change can break integration coverage for a completely different area that its own unit/widget tests never touch. A normal single-feature ticket has a narrower, more contained blast radius, so the existing gates (widget tests + review) are sufficient there. Not yet turned into a workflow-doc rule — capturing the scoped takeaway here for now.

## Debrief summary

### 2026-07-24

**What went well**
- All 5 WUs shipped cleanly against the approved plan with essentially no deviations — the plan's exact helper signatures and call-site designs (WU4's `_autoFailShowup`, WU5's `_mergeRemoteRecord<T>`) were specified precisely enough that implementation matched them almost verbatim.
- Architectural and runtime/migration reviews were consistently clean across all 5 PRs plus the hotfix — no real findings, only one non-blocking cosmetic nitpick (WU2's redundant `isA<FilledButton>()` assertion, WU4's skipped job-comment number).
- The "## Work Units" checklist convention (carried over from HAB-187) made cross-session tracking of a 5-WU ticket painless — always clear what shipped and what remained.

**What was hard or surprising**
- WU3's icon-family fix broke two integration tests (`sync_on_login_flow_test.dart`, `fresh_install_flow_test.dart`) that had the same Material/Cupertino bug baked into their own hardcoded assertions — invisible to `flutter test` since integration tests need a device, so it only surfaced when scenarios were run manually well after WU3 had merged.
- A `ship` subagent (haiku) hit its session limit mid-task on WU5 and had to be finished manually — worth noting `ship` runs can fail silently partway through with no actual writes applied; always verify Linear/git state directly rather than trusting a subagent's completion report at face value (this also happened once on WU2's ship, where the agent claimed Linear was updated but hadn't touched it).

**What to change**
- For checkup-cleanup-style tickets specifically (independent WUs bundled together, touching cross-cutting/unrelated areas of the app) — run `/run-scenarios` or the affected integration test files before merging each WU, not just unit/widget tests. Not a blanket rule for every ticket; scope it to this ticket shape, where one WU's fix can silently break integration coverage for a completely different area.
- Continue verifying subagent-reported Linear/git writes against the actual source (Linear API, `gh pr view`) rather than trusting the completion summary — this caught two real discrepancies in this ticket alone.
