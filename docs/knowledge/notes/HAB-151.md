### 2026-07-06

**run-scenarios fails deterministically, not flaky ÔÇö ~30/53 scenarios failing identically across two reruns**

Checked the two most recent `workflow_dispatch` runs of `run-scenarios` on `feature/HAB-151-ci-run-scenarios-android`, both against the same commit (`e868c40`, so this is a clean rerun of identical code, not drift). The same set of tests fails both times, at the same points ÔÇö this rules out ordinary flakiness.

Two distinct problems layered on top of each other:

1. **Real-timing waitFor failures.** Several unrelated test files (`stop_pact_flow_test`, `showup_to_pact_navigation_flow_test`, edit-pact flows, archive-pact flows, `pact_note_flow_test`) each independently call `waitFor(tester, find.text(strings.stopPact))` or wait on the `pact-note-field` key to confirm the Pact Detail screen finished loading. `waitFor`'s timeout is 10s (`integration_test/harness.dart:224`). On the real API-29 CI emulator these consistently report "Found 0 widgets" ÔÇö the screen's key content isn't rendering within 10s of real wall-clock time. This is new because `run-scenarios` is the first place this suite runs against real timing; previously it only ran on host/widget-test timers, which don't block on real async delays the same way. Root cause not yet identified ÔÇö could be a genuinely slow load path in `PactDetailViewModel`, CI emulator resource starvation, or the 10s timeout simply being too short for a real device.

2. **Cascade after the first hard failure.** Once one test throws an uncaught `TestFailure` (e.g. test 47 in `pact_note_flow_test`), every subsequent test in the same run (tests 48 through `tearDownAll`, ~26 tests) is marked `error` with no individual diagnostic ÔÇö the shared app instance/isolate is left in a bad state. This inflates the apparent failure count and gives no signal on those later tests.

**Branch rebased onto latest `origin/main`**

`feature/HAB-151-ci-run-scenarios-android` was 10 commits ahead of its base and out of date after PR #238 (Russian plural-form fix) merged. Rebased onto `origin/main` and force-pushed. Conflicts only in `docs/CHANGELOG.md` and `pubspec.yaml` ÔÇö both resolved by keeping both changelog entries (HAB-151's `[0.50.0]` placed above the merged `[0.49.6]`) and bumping the version to `0.50.0+156`.

**Next steps**

- Investigate why the Stop Pact button / note field is slow to render on-device: genuine app bug vs. CI emulator resource constraint vs. timeout too short.
- Prevent one hard test failure from poisoning the rest of the suite ÔÇö e.g. reset/relaunch app state between tests, or isolate failures so later tests still get a fair run.

### 2026-07-07

**Android run-scenarios job: 3-4 tests failing, likely enterText()/onChanged reliability on the AVD, not just emulator slowness**

Repeated failures on `feature/HAB-151-ci-run-scenarios-android` (e.g. runs 28845612104, 28843739652, and several earlier attempts). All failures are in `edit_pact_flow_test.dart` and `pact_note_flow_test.dart`, and all share the same pattern: `tester.tap()` + `tester.pumpAndSettle()` to focus a text field, then `tester.enterText(...)`.

- On the Android AVD (KVM-backed `ubuntu-latest` runner), `enterText()`'s simulated keystroke does not reliably fire the field's `onChanged` callback. The diagnostic `expect()` added in commit `d1df9cd` ("test(scenarios): add diagnostics + remove pre-tap for edit_pact failures") ÔÇö `expect(find.text('Morning Run'), findsWidgets, reason: 'enterText should update AppBar title via setHabitName')` ÔÇö itself fails, finding 0 widgets.
- This is **not** a plain slow-emulator/timeout issue: `waitFor`'s default timeout on this branch was already bumped from 10s to 90s in `harness.dart`, and the failures still occur.

Failing tests observed:
1. **"renaming pact from pact detail updates the displayed habit name"** ÔÇö the enterText diagnostic assertion fails before save even happens.
2. **"renaming pact updates the habit name shown on ShowupDetailScreen"** ÔÇö save succeeds with the correct new value (confirmed via a `repo.habitName` print and the `pact_edit_saved` analytics event firing), but `ShowupDetailScreen` still shows the old name after navigating back. This matches the test's own code comment describing a suspected pre-existing bug: `ShowupDetailViewModel.habitName` is captured at initial load and never refreshed after a rename elsewhere.
3. **"existing stop reason is pre-populated; editing and saving persists the new note"** and 4. **"empty note field is shown; writing and saving persists the note"** ÔÇö both persist the OLD/stale note value (or `null`) instead of the newly typed one ÔÇö consistent with the same enterText/onChanged reliability problem, this time in `pact_note_flow_test.dart`.

Context: prior commits on this branch already fixed a long list of other Android-specific flakiness (swipe offsets, off-screen taps, scrollable ancestors, single-APK test runner, timeout bumps), so this looks like the last blocker before the job is stable. Worth investigating next:
- Whether `tester.enterText()` needs an extra `pump()`/`pumpAndSettle()` cycle after focus is established before the platform `TextInputClient` connection is actually ready to deliver the edit on this specific Android emulator image (cold-JIT / KVM timing).
- Separately, whether `ShowupDetailViewModel` actually re-reads the pact's `habitName` on return navigation or only captures it once at construction ÔÇö that looks like a real product bug independent of the CI flakiness.

### 2026-07-07

**Process reflection: reaching CI-green took a lot of guessing, not the usual systematic approach ÔÇö account for this on future bring-up tickets**

End state is good ÔÇö most scenarios pass on CI now, and the full suite passes locally. But getting there was a long trial-and-error loop, closer to guessing at root cause than the systematic, hypothesis-first approach used on other tickets. This ticket seems to be a special case: it's "set this up once and forget about it" work ÔÇö standing up integration scenarios against a real Android emulator for the first time ÔÇö which surfaces a long tail of environment-specific issues that nothing else would have caught. For future tickets of this shape (new CI platform, new emulator target, first-time real-device timing), expect that many "underwater stones" up front rather than treating it as routine wiring. Full debrief (and any resulting workflow/skill changes) deferred until the implementation is done and all scenarios pass stably.
