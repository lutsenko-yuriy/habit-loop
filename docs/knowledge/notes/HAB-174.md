# HAB-174: Populate a shared pact-detail cache (stats + timeline) when Pact Details is first opened

## Notes

### 2026-07-16

**Process safeguards reduce dead code but don't eliminate it — stay careful during implementation and review**

While planning this ticket, asking to double-check WU0's scenario coverage for the "notes get saved and shown" write-through path surfaced a real bug: `PactService.updatePact` only pushed to the shared cache on the `completed`-status branch, so a plain note edit would silently never refresh the cache (the new scenario would've failed once built, if it hadn't been added at plan time). Regular audits, reviews, checkups, small PRs, and rigorous unit/scenario testing all help a lot during development — but this slipped through the initial plan anyway. Worth staying deliberately careful during implementation and code review rather than assuming the process nets cover everything.

### 2026-07-16 — WU scope summary (approved plan)

- **WU0** — integration scenarios only, no production code. Two scenarios: (1) mark a showup done on Pact Details → open Timeline same session, no restart (status write-through); (2) edit the pact-level note on a stopped/completed pact's Pact Details → open Timeline same session, `PactConcludedMilestone.note` reflects the edit (note write-through, extends the existing `_stoppedPact`/`_completedPact` fixtures in `pact_note_flow_test.dart`).
- **WU1** — new `PactDetailBundle` + `PactDetailCache` (`lib/slices/pact/application/`), built and unit-tested in isolation against fakes, including a golden-equivalence test vs. the existing unmodified `PactStats.compute`/`PactTimelineGrouper.group`. No consumer wired yet. Single-pass computation attempted first per this session's explicit override of the HAB-126-carried-forward "don't fuse the loops" call; two-loop fallback documented and needs no separate sign-off if the fusion proves messy.
- **WU2** — wires `PactStatsService`, `PactService`, and `PactDetailViewModel` onto `PactDetailCache`; deletes `_statsCache` and `PactStatsService.onPactCompleted` (superseded by `PactService.updatePact`'s unconditional `cache.refresh` call — the fix for the note-write-through gap found above). Makes both WU0 scenarios green.
- **WU3** — wires `PactTimelineViewModel` onto the same cache; deletes `PactTimelineCache`/`PactTimelineService` and their tests; drops the HAB-173 `initialHabitName` forwarding in favor of `cache.peek()`; corrects the stale `docs/ARCHITECTURE.md` claim that `persistShowupStatus` used to evict `PactTimelineCache` (it never did in the current code).

Full plan text and rationale: see the two plan comments on the HAB-174 Linear issue (initial plan + the WU0/PactService follow-up).

### 2026-07-16 — WU0 gained a third scenario (showup-note write-through)

While reviewing WU0's scenarios, spotted that scenario 2 (note write-through) only covered the **pact**-level note (`stopReason`), not a **showup**-level note — an easy mix-up since both are called "note" in casual conversation. Checked the existing suite for showup-note UI coverage and found none: every existing scenario touching a showup note (`redeem_showup_flow_test.dart`, the noted-showup case in `pact_timeline_flow_test.dart`) seeds the note directly into the repo as a fixture — none of them type into the note field and tap save. The note `TextField`/save button on Showup Detail (`showup_detail_page_android.dart` / `_ios.dart`) also have no test `Key`s at all, which is presumably why. Added a third WU0 stub, `showup_note_write_through_visible_in_timeline_same_session`, alongside the other two in `pact_timeline_flow_test.dart`'s new write-through group — with a TODO note for the implementer to add the missing `showup-note-field` / `showup-note-save-button` Keys as part of building the driver (small, behavior-preserving addition, not new production logic). Kept out of WU0 itself since WU0 is scenarios-only, no production code.
