# HAB-153: Audit and clean up integration test scenarios

## Notes

### 2026-07-08

**iOS-only onboarding carousel flake root-caused and fixed (WU0)**

The one failing iOS scenario was "Onboarding carousel flow / swiping right returns to slide 0" in `integration_test/onboarding_carousel_flow_test.dart`. Root cause: unlike the sibling "swiping left advances to slide 1" test, this test never called `tester.pumpAndSettle()` after the first `waitFor(slide1Title)` before firing the second drag back to slide 0. On the iOS Simulator's 402pt-wide screen, the still-mid-transition slide 1 heading's computed drag-center landed at x=403 — 1px past the render bounds — so the return drag never hit-tested, and the trailing `waitFor(slide0Title)` timed out. Android's different screen metrics happened not to expose this. Fixed by adding the missing `pumpAndSettle()` call (test-only, one line). Verified: full suite now passes 59/59 on both iOS Simulator (iPhone 17) and Android emulator (Pixel 5 API36).

**WU2 shipped: harness helper extraction (mechanical)**

Moved `openPactsPanel`, `openPactDetail`, `openTimeline` (the more defensive variant — waits for the timeline button before `ensureVisible`), and a shared `noAutoAdvance` RC override into `harness.dart`, replacing 4 duplicated definitions across up to 8 files. Both architecture and audit review passes independently flagged the same nuance: `redeem_showup_flow_test.dart`'s original `openTimeline` lacked the defensive wait that the shared version now applies to its 4 call sites — a disclosed, intentional widening (not a coverage change), confirmed low-risk by both reviewers. Net -62 LoC, 59/59 tests green on both iOS and Android. User feedback for debrief: PRs that shrink total LoC while preserving coverage are a good signal worth calling out explicitly when they happen.

**Process gap: WU scope got re-explained after being lost to compaction**

After context compaction, the user asked what "fixture builders" (WU3) meant — the definition had already been given during planning, but wasn't retained in a form that survived compaction. User feedback (self-directed, for debrief): when a multi-WU plan is approved, generate a small ticket-scoped summary (one line per WU: what it touches, why) and refer back to it, rather than relying on the full plan discussion staying in context. This would make WU handoffs resilient to compaction instead of requiring the user to re-ask.

**WU3 shipped: fixture builders (mechanical)**

Added `buildPact`/`buildShowup` to `harness.dart` with sensible defaults (far-future dates, `DailySchedule(hours: 8)`, `PactStatus.active`/`ShowupStatus.pending`, 10-minute duration) and, notably, `createdAt` defaulting to `startDate` — every one of the 22 pre-existing `Pact(...)` literals across the suite already set `createdAt == startDate`, so this single default eliminated that field from every call site. The Android run hit one transient ADB/emulator disconnect ("log reader stopped unexpectedly") mid-session — unrelated to the code change, resolved by an `adb` daemon restart and a clean retry.

The WU1 findings report (finding 5) claimed `edit_pact_flow_test.dart` and `pact_note_flow_test.dart` already used the `snake_case_name: description` `testWidgets` naming convention. Re-verifying from source at the start of WU4 showed this was wrong — only `archive_pact_flow_test.dart`, `pact_timeline_flow_test.dart`, and `redeem_showup_flow_test.dart` actually conform. Lesson: audit findings that name specific files as "already conformant" should be spot-checked against source before the next WU treats them as given, especially across a compaction boundary.

### 2026-07-09

**Process gap: a silent monitor gave no signal that a task had actually finished**

During WU3, the first background-test monitor (waiting for a single "All tests passed" line) ran for its full timeout with zero output and no notification, even though the underlying `flutter test` run had already finished (with an ADB/emulator error) well before the timeout. The gap was only noticed after the user asked directly whether there was any progress. A monitor that emits nothing is indistinguishable from "still running" — it should be built to also emit on completion/error, not just on a single success marker, so silence never has to be resolved by the user asking.

## Debrief summary

### 2026-07-09

**What went well**
- Overall the ticket went extremely well — user very satisfied.
- Despite being a big multi-WU ticket (5 WUs), token spend (mostly Linear-side) came in lower than usual for tickets of this size. Attributed, tentatively, to the WU-by-WU structure — user isn't fully certain but flagged it as the most likely driver.
