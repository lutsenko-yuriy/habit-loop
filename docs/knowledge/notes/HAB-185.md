# HAB-185: Gate pubspec.yaml version bumps on app-changing CHANGELOG entries

## Notes

### 2026-07-19

**WU scope summary** (per approved plan, comment on HAB-185):

- **WU1 — Conditional bump + `## [Unreleased]` section** (`feature/HAB-185-WU1-conditional-bump`, ~70 LoC, docs/skill only). Edits `skills/manage/ship/SKILL.md` (step 2: route non-`[user]`/`[app]` entries to a permanent `## [Unreleased]` bucket instead of a numbered heading; step 4: bump `pubspec.yaml` only when the entry has `[user]`/`[app]`). Scaffolds `## [Unreleased]` in `docs/CHANGELOG.md`. Updates `docs/VERSIONING.md`, `docs/FEATURE_WORKFLOW.md` step 7, and `AGENTS.md`'s versioning line. No unit tests (prose/skill instructions) — validated by a manual `ship` dry-run on both paths.
- **WU2 — `lint.py` Unreleased tag-hygiene guard** (`feature/HAB-185-WU2-lint-unreleased`, ~80 LoC). Adds a scan of the `## [Unreleased]` block to `scripts/changelog/lint.py` that rejects unknown bracket tags (reusing `KNOWN_TAGS`) — without it, a typo'd tag there goes uncaught since lint's existing regex only matches numbered `## [X.Y.Z]` headings. Adds corresponding `test_lint.py` cases.

Key plan finding: `distribute.py`/`release_notes.py`/`lint.py` all key off a numeric-version regex, so `## [Unreleased]` (non-numeric) is silently skipped by all three already — no functional change needed to the "which entries are new since last tag" logic. Critical invariant: `## [Unreleased]` must always stay at position 0 in the file; new numbered headings insert below it, never above.

### 2026-07-19 (later)

**Design pivot: single permanent bucket → interspersed, bounded `## [Unreleased]` sections.** After WU1's first review pass, user feedback: a single permanent `## [Unreleased]` at position 0 would grow forever, forcing a long scroll to reach recent numbered releases. Revised design: `## [Unreleased]` batches are bounded — sealed in place between two numbered releases once the next app-changing entry ships above them; a fresh one opens at the top for the next batch. This invalidated the "no functional change needed" finding above: `distribute.py`/`release_notes.py`/`lint.py`'s body-boundary logic (next *numeric* heading) would silently absorb a sealed Unreleased batch into the newer release's body. Fixed via a new shared `scripts/changelog/heading_boundaries.py` helper (bounds on *any* `## [...]` heading) consumed by all three. Full detail in the Linear comment thread on HAB-185.

**Process lesson (user's own observation, for debrief):** "All three alternatives proposed by the brief considered that the unreleased stuff goes either into a separate section outside of the list or is a part of one of the numbered releases. I should have read the proposed stuff more carefully and not take whatever was recommended first." — the `/brief` dialog's 3-option table (Unreleased bucket / shared pending heading / fold into next release) never actually included "bounded, interspersed Unreleased sections" as an option; the user picked the first/recommended one without fully vetting whether it matched what they actually wanted, and the gap only surfaced once looking at the real implementation. Worth surfacing explicitly in the HAB-185 debrief as a "what to change": scrutinize proposed options harder before picking, and/or the option-proposer (me) should try to more exhaustively enumerate the real design space rather than anchoring on the most obvious framing.
