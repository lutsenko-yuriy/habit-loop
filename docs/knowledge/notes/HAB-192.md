## Debrief summary

### 2026-07-23

**What went well**
- Fast turnaround: reproduced, root-caused (a variable-length PCRE lookbehind that could never have matched), and fixed in one sitting.

**What was hard or surprising**
- The bug was latent since HAB-183 WU1 (2026-07-19) and only surfaced now because every intervening `main` push was an HAB-187 `[wip]` WU, which skips the `resolve-version` job entirely.
- No manual-dispatch workflow exercises the real `resolve-version` composite action's production code path — `build.yml` deliberately reimplements a trivial stub instead of calling the composite (to avoid consuming a build number), and `test.yml`/`scenarios.yml` don't touch it at all. The only way to validate a fix to this path is a real qualifying push to `main`.

**What to change**
- Would like a dry-run alternative for validating composite-action logic (not just `resolve-version` — the same gap applies to `build-android-app`, `build-ios-app`, etc.) without needing an actual `main` push. Captured as a new backlog ticket (HAB-193) rather than designed here — likely to sit in the backlog for a while, but worth having on record.
